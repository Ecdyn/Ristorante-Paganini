# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3: German Onepage — Head and SEO Skeleton.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> Phase 3 is a structural and SEO phase. It produces the /de/index.html document
> with a complete `<head>` section and an empty `<body>` skeleton — no visible content,
> no images, no new interactive states. All design tokens were locked in Phase 1
> (assets/css/main.css). This spec documents inherited tokens, defines the structural
> markup contract, and establishes the meta/SEO copywriting contract.

## Structural Markup Contract

Phase 3's primary output is document structure. This section defines the exact markup
requirements that the executor must satisfy.

### Document Root

```html
<!DOCTYPE html>
<html lang="de">
```

- `lang="de"` on html element — required for FOUND-01, LANG-01
- Exactly one `<h1>` in the skeleton — the restaurant name in the header/hero area
- Phase 3 places a placeholder `<h1>` comment; Phase 4 fills it

### Required `<head>` Elements (in order)

1. `<meta charset="UTF-8">`
2. `<meta name="viewport" content="width=device-width, initial-scale=1.0">`
3. Meta title: `<meta name="title">` and `<title>` (German, see Copywriting Contract)
4. Meta description (German, see Copywriting Contract)
5. Canonical: `<link rel="canonical" href="https://www.example.com/de/">` with `<!-- PLACEHOLDER: Replace example.com with real domain before launch -->` comment
6. Hreflang block — all 4 tags (see Hreflang Contract below)
7. Open Graph tags — full set (see OG Contract below)
8. Twitter card tags (see Twitter Contract below)
9. Favicon link tags — exact 4 tags from Phase 2 (02-02-SUMMARY.md), using `../assets/` relative prefix
10. `<link rel="stylesheet" href="../assets/css/main.css">`
11. JSON-LD script block (see JSON-LD Contract below)
12. `<script src="../assets/js/main.js" defer></script>`

### Hreflang Contract (D-12, D-13, D-14, SEO-03)

All four tags required on this page — reciprocal cluster begins here:


## Implementation Notes for Executor

1. **Asset path prefix:** All `../assets/` references use relative paths from `/de/index.html`. Consistent with Phase 2 convention documented in 02-02-SUMMARY.md.

2. **Header/nav markup source:** Adapt `assets/css/test.html` (Phase 2) for the real `<header>` in `/de/index.html`. Reuse `id="site-header"`, `id="menu-toggle"`, `id="mobile-menu"` — these are exact JS hook names in main.js.

3. **Hreflang base URL:** Use `https://www.example.com` as placeholder. Wrap the entire hreflang block in a `<!-- PLACEHOLDER: Replace example.com with real domain before launch -->` comment. Same placeholder format in canonical and OG tags.

4. **JSON-LD placement:** Place `<script type="application/ld+json">` in `<head>`, before the closing `</head>` tag, after the CSS and JS links. Google parses head JSON-LD reliably.

5. **Single H1 rule:** Phase 3 places the `<h1>` in the skeleton as an empty placeholder with a comment, e.g., `<h1><!-- EDITABLE: Restaurant name — e.g., "Ristorante Paganini" --></h1>`. Phase 4 fills it. Never add a second H1.

6. **No robots meta tag needed:** Omitting `<meta name="robots">` defaults to `index, follow` — no need to declare it explicitly. Only legal pages need `noindex` (Phase 6).

7. **Aria-label on landmark regions:** Whether to add `aria-label` attributes to `<section>` elements is left to Claude's discretion (D-03 in CONTEXT.md). Recommendation: add `aria-labelledby` in Phase 4 when heading IDs exist, not in Phase 3 skeleton.

8. **FAQPage JSON-LD is Phase 4 scope:** Do not add FAQPage schema to the JSON-LD block in Phase 3. The `#faq` section is created as an empty element only (D-05).

9. **No Google Fonts CDN link:** Self-hosted fonts only. No `<link rel="preconnect" href="https://fonts.googleapis.com">` or similar. GDPR requirement (CLAUDE.md, STATE.md Init decisions).
